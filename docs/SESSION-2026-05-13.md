## 错误清单

### 1. CI 失败：ruff lint 错误
- **现象**: GitHub Actions Run #33 在 Python 3.11 矩阵失败，错误步骤 `Run ruff check`
- **原因**: 新写的 4 个覆盖率补齐测试文件中存在未使用的 import（F401：pytest、uuid、concurrent.futures、MagicMock）和未使用的变量（F841）
- **影响**: 所有 5 个 Python 版本的 CI 作业均被取消，没有执行到测试阶段
- **修复**: `ruff check tests/ src/ --fix` 自动清理 9 个错误，提交 `370b945`
- **根本原因**: 缺少 "提交前必须运行 ruff" 的流程约束

### 2. 测试颠测性失败：adaptive_tuner ε-greedy
- **现象**: `test_auto_tune_numeric_no_change` 在本地单独运行通过，但在批量运行时偶尔失败，报错 `assert 'p1' not in changes`
- **原因**: `_tune_numeric_parameter` 中 `random.random() < 0.1` 触发了 ε-greedy 探索，返回了随机值（如 12）而非基准值（10）
- **影响**: 测试结果非确定性，CI 可能偶尔失败
- **修复**: 在测试中 patch `random.random` 返回 0.5，跳过 ε-greedy 分支，提交 `e1b5541`
- **根本原因**: 缺少对随机性源的 patch，依赖了确定性行为

### 3. GitHub release 创建工具缺失
- **现象**: `brew install gh` 超时180秒未完成
- **原因**: 环境中没有预装 `gh` CLI
- **解决**: 转而使用 GitHub REST API + `urllib.request` 直接创建 release，成功发布 v0.6.1
- **经验**: API 调用比 CLI 工具更可靠，不依赖环境安装

### 4. 内存匹配失败
- **现象**: `memory(action='replace')` 多次报错 "No entry matched"
- **原因**: old_text 中包含中文标点和空格，与存储的文本完全匹配失败
- **解决**: 改用 `action='add'` 直接追加新条目，而非替换
- **经验**: memory 的 replace 应使用极简短语作为 old_text，避免完整句子匹配

## 经验总结

### 测试编写
1. **涉及 random 的测试必须 patch**: 任何依赖 `random.random()`、`random.randint()` 的代码，测试时必须用 `unittest.mock.patch` 控制，确保确定性
2. **批量写测试时先看清源码**: 不要假设 API 签名，用 `inspect.signature()` 确认后再写
3. **测试类名不要跨文件重复**: 之前 `TestAdaptiveTuner` 在多个文件中重复，导致 pytest 收集异常
4. **模拟 I/O 而非真实调用**: AlertChannel 测试用 `unittest.mock.patch` 模拟 urllib 和 smtplib，避免网络调用

### CI 与代码质量
5. **提交前必须三件套**: `ruff check` + 全量测试 + 覆盖率检查
6. **ruff 自动修复可信**: `ruff check --fix` 能处理 90% 以上的样式问题，但仍需人工复审
7. **CI 红了先看第一个失败步骤**: Run #33 失败步骤是 `Run ruff check`，而非测试本身失败，这说明是样式问题

### 工具链
8. **gh CLI 不可靠时用 API 替代**: `brew install` 可能超时或失败，GitHub REST API 更稳定
9. **SQLite credential 存储比环境变量安全**: 从加密的本地数据库读取 token，不在 shell history 中暴露

### 文档与索引
10. **多语言 README 必须同步**: 修改了中文 README 后，en/es/fr/ja/ko 版本的 badge 也需同步更新
11. **CONTRIBUTING.md 是防止 CI 红的最佳场所**: 在其中明确写入 "提交前必须运行 ruff 检查"，可避免未来贡献者重复犯错
